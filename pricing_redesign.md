## 1. Opus 전략 — Premium 전용 + 일일 한도

### 결정
- Opus 토글은 **Premium 사용자만** 활성화 가능
- Premium 안에서도 **일일 30번** 한도 (메인 대화에서만 사용)
- 새벽 4시 리셋 (기존 일일 chat cap과 일관)
- 분석 / 마법고동 / 리뷰는 기존대로 (대화가 아닌 영역)

### 근거

**(a) 가격 사다리에 의미 부여.** 기존엔 Light/Premium 차이가 *양*뿐. 추가팩으로 회피 가능 → "충전 폐기" 의도 무력화. **기능 차별화**가 들어와야 Premium이 진짜 의사결정이 됨.

**(b) Light 사용자의 Opus 5x 차감 시나리오 차단.** 운영 안정성 ↑.

**(c) 업그레이드 동인의 정서 전환.** "양이 부족해서 → Premium" (쪼들림, 부정적) → "Opus 더 쓰고 싶어서 → Premium" (끌림, 긍정적). 같은 전환인데 사용자 감정 정반대.

**(d) Premium 내부 cap 도달 우려 해결.** Opus 무제한이면 헤비 유저가 cap 빠르게 소진 → "마음껏 깊게" 약속 깨짐. 일일 30번 한도면 평균 사용자는 의식 안 함, 어뷰저성 사용만 차단.

### 일일 한도 = 30번의 근거
- 자기관찰 앱 사용 패턴: 하루 5-15분, 평균 대화 10-30턴
- 깊은 대화는 그 중 일부 (3-10턴 자연스러움)
- 30번 = 헤비 사용자도 만족, cap 도달 거의 X
- 시뮬레이션: 30 Opus 턴/일 × 30일 ≈ $7.5 → cap $13 안에서 안전

### 한도 도달 카피
> **"오늘 깊은 대화 다 나눴네. 이만 여기까지 하고 쉬자 🫂"**

- 🐚 (앱 마스코트) X, 🫂 (순간의 정서) O
- "내일 또" 같은 미래 약속 X — 현재로 닫음
- 자기관찰 앱의 **충분함의 감각** 정서

### 한도 노출 정책
- 평소 카운터 **숨김** (토큰제 멘탈모델 회피)
- **20번 도달부터** 부드러운 알림 ("Opus 깊은 대화 10번 남았어")
- **30번 도달 시** 위 카피 모달

### 튜토리얼에서의 Opus
첫 시작 코어 튜토리얼 동안에는 사용자가 Opus 체험 가능. 카피:

> "지금은 튜토리얼이니까 Opus 써볼 수 있게 해줄게"

체험 후 Premium 업그레이드 동인 자연스럽게 형성. **체험 없이 Premium 전용이라고만 하면 사용자가 Opus 가치 모름.**

### 코드 변경 메모
```javascript
// state 또는 user 객체에 plan 정보 확인
function canUseOpus() {
  if (state.tutorial?.inProgress) return true;  // 튜토리얼 예외
  if (user.plan !== 'premium') return false;
  return user.opus_daily_used < 30;  // 일일 한도
}

// 새벽 4시 리셋 (기존 dailyChatCap 패턴 재활용)
// opus_daily_used: number, opus_daily_reset_at: timestamp
```

## 2. Light / Premium 가격 조정

### 결정

| 플랜 | 기존 | 신규 |
|---|---|---|
| Light | 8,900원, cap $5 | **9,900원, cap $5 유지** |
| Premium | 25,000원, cap $15 | **25,000원, cap $13** |

### 근거

**Light 9,900원**:
- 만 원 미만 심리적 가격대 유지 (8,900 → 9,900은 +11.2%, 사용자 체감 거의 없음)
- 마진 약 28% → 부가세/PG 수수료 흡수 후 실질 18-20%
- "거의 만 원" 심리 대신 **"만 원 안쪽"** 보존

**Premium 25,000원, cap $13**:
- 가격은 그대로 (챗GPT Pro 22,000원 라인 근처 포지셔닝)
- cap $15 → $13 인하: 마진 16% → 27%, 인프라/세금 흡수 가능
- $10까지 내리면 cap 도달률 급증 → "마음껏 깊게" 약속 깨짐 → $13이 절충
- Opus 일일 30번 한도와 결합되어 cap 보호 이중 안전망

### Light 워딩 변경
기존: ~~"가볍게 매일 / 짧은 대화 / 간단 분석"~~
신규: **"매일의 자기관찰. 일반 대화 + 분석 풀로."**

근거: "가벼운 / 짧은 / 간단" 워딩은 *Light = 부족한 것*이라는 인상 → Premium upsell엔 도움이지만 부정적 동인이라 본인 톤과 안 맞음. **Light는 결핍이 아닌 그 자체로 완전한 무언가로 정의.** Premium은 *그 위에 더 있는 것*.

## 3. 얼리 유저 플랜 — 4,900원/월

### 결정
- **가격**: 4,900원/월 (Light 9,900원의 약 50%)
- **cap**: $4 (≈ 5,600원치) — Light cap $5의 80%
- **자격**: 출시 컷오프 날짜 *이전* 가입한 사용자, 해당 계정에 한정
- **유효 기간**: **평생**, 한정 인원 제약 X
- **Opus**: 사용 불가 (Light와 동일, Premium에서만)
- **추가팩**: 사용 불가 (Premium 전용)
- **Premium 결제 시**: 정가 25,000원 (얼리는 Light 자리만)
- **Premium → 다운그레이드**: 자동 4,900원 복귀 (early_user flag 영구)

### 근거

**(a) 진짜 감사 표시.** 출시 *전*에 함께해준 친구·가족·초기 사용자에 대한 영구 가치. *"한정 기간 할인"* 같은 마케팅 도구 X. 본인 톤 = 정직, 평생 = 정직의 표현.

**(b) 50% 할인의 임팩트.** 마케팅 심리학에서 "반값"은 30%/40%와 *질적으로* 다름. "절반 가격" 한 마디로 이해 가능.

**(c) cap $4의 미세 적자 감수.** 마진: 4,900 - 5,600 = **-700원/사용자**. 옵션 (d) 절충 — 양 줄여서 적자 줄임. 평균 사용자 ($2.5 사용 추정)는 cap 도달 거의 X. 헤비 사용자만 Premium 전환 권유.

**(d) Premium 업그레이드 동인 보존.** 4,900 → 25,000 = 5배. 얼리 유저가 Opus 원하면 Premium 결제 가능.

## 4. 추가팩 — Premium 전용

### 결정
- `light_pack` (5,000원, +$4) **제거**
- `premium_pack` (7,000원, +$5) **유지**
- Light/얼리 사용자는 cap 도달 시 Premium 전환 또는 다음 달 대기

### 근거

**(a) Premium 위상 보호.** 추가팩이 모든 tier에서 가능하면 "Premium ≈ Light + 추가팩"으로 인식 → Premium의 카테고리적 차별성 흐려짐. 추가팩 = Premium 전용이면 Premium은 *마음껏 쓰는 자리*로 명확.

**(b) 의사결정 단순화.** Light/얼리의 cap 도달 시 "추가팩 살까 Premium 갈까" 미세 의사결정 X. 선택지: **Premium 또는 다음 달 대기**.

**(c) 헤비 사용자 → Premium 자연 funnel.** Light/얼리에서 cap 자주 도달 = Premium 적합 사용자.

**(d) 운영 단순화.** 결제 종류 3개 → 2개. 코드 단순.

### Cap 도달 시 UI 분기

| 사용자 | 표시 |
|---|---|
| 얼리 / Light | "이번 달 한도 도달. Premium으로 가면 더 깊게 쓸 수 있어. 다음 달까지 기다려도 OK." |
| Premium | "추가팩 7,000원으로 더 쓸까? (1회성)" |

**얼리/Light 카피의 핵심**: "다음 달까지 기다려도 OK" — 강매 X. 본인 톤 보존.

### Premium → Light 다운그레이드 안내
Premium 해지 시 모달 명시:
> "다음 달부터 Light로 변경돼. 추가팩은 Premium에서만 가능하니, Light에서 한도 도달하면 다음 결제일까지 기다려야 해."

## 5. 환영 선물 — 정확히 100만 토큰

### 결정
- **양**: 정확히 100만 토큰 (cap을 100만 토큰으로 hard cap)
- **시점**: 첫 시작 코어 튜토리얼 *완주 후* grant
- **튜토리얼 진행 중 비용**: 회사 부담 (사용자 잔액 차감 X)
- **만료**: **30일**, 7일 전 알림
- **서비스 제공 모델**: Sonnet only (Opus는 튜토리얼 중에만)

### 근거

**(a) 정확히 100만 = 정직 톤.** "약 100만"의 *"약"*은 *under-promise, over-deliver* — 영업 마인드셋. 본인 앱 정체성은 *직설적 정직*. 보험 X.

**(b) 튜토리얼 후 grant의 4가지 의미**:
1. **어뷰저 차단**: 튜토리얼 완주 = 성의 검증. 일회용 이메일로 봉사 후 grant 빠는 시나리오 차단.
2. **가치 인식 선행**: 사용자가 앱이 뭔지 알게 된 *후*의 grant는 진짜 invitation.
3. **튜토리얼 비용 = 서비스 소개**: 사용자 보너스 영역과 분리. 영역 분리가 자연스러움.
4. **데이터 노이즈 제거**: 튜토리얼 중간 이탈자 grant 받지 않음 → retention 지표 깔끔.

**(c) 100만 토큰 산출 (Sonnet 4.6 기준)**:
- input $3 / 1M, output $15 / 1M
- 캐시 적용 (90% 할인) 시 평균 사용자 비용: $1.5-2 (≈ 2,100~2,800원)
- 본인 기존 3,000원 예산 대비 가성비 우수
- 토큰 수치 자체가 *큰 숫자* 임팩트

### 워딩 (3-stack)
```
환영 선물 · 무료 체험
🐚 100만 토큰
약 한 달치 자유로운 대화
```

세 줄 각자 다른 역할:
- "100만" → 임팩트 (감정)
- "토큰" → 신뢰 (기술적 정확성)
- "한 달치 대화" → 가늠 (실용)

### 튜토리얼 내 Opus 카피 (변경 사항)
기존: ~~"고동MOM이 무료 토큰 넣어줬으니까..."~~
신규: **"지금은 튜토리얼이니까 Opus 써볼 수 있게 해줄게"**

근거: "고동MOM이 토큰 넣어줬다"는 *어색한 sponsorship* 톤. 튜토리얼 단계의 *체험 제공* 톤이 자연스럽고, Premium 업그레이드 동인과도 연결.

### 만료 정책
- grant 시점 30일 후 자동 만료
- **만료 7일 전 알림** (다중 채널 — 6번 섹션 참조)
- 알림 톤: "사라져" 같은 다급한 워딩 X, "환영 선물이 한 달 동안 유효해서 알려드려" 부드럽게

## 6. 결제 알림 채널 — PWA 토스트 단독 X

### 결정
- 토스트 알림은 휘발성 → **단독 사용 X**
- 다중 채널 + 영구 기록 필수
- **포트원 가입 시 카카오 알림톡 채널 함께 신청**

### 근거 (한국 정기결제 사전고지 규정)
- 콘텐츠산업진흥법 / 표준약관: 유료 전환 시점 기준 최소 7일 전 서면, 음성전화, 문자 등으로 통지
- 한국소비자원이 **단순 이메일 통지**도 *알기 어려운 방식*으로 지목 — 강하게 단속 중
- 가입 시 사전 고지 1회는 부족, 7일 전 *재안내* 의무
- PWA 토스트 = 영구 기록 X, 휘발성 → 통지 의무 단독 충족 어려움

### 다중 채널 디자인

**현 단계 (자동결제 X, 환영 선물 만료 알림만)**:
1. 앱 헤더 **상시 배너** ("환영 선물 7일 남았어") — 토스트 X, 영구 표시
2. 앱 안 **알림 인박스** (영구 기록)
3. **이메일 통지** (Supabase auth 이메일)

**포트원 도입 후 (자동결제)**:
1. **카카오 알림톡** (한국 표준약관 인정 채널, 도달률 90%+)
2. **이메일 통지** (영구 기록, 전자상거래법 §8 영수증 의무)
3. **앱 인박스** (앱 내 영구 기록)

### 카카오 알림톡 도입 메모
- **개설 조건**: 사업자등록증 (본인 이미 보유 ✅)
- **비용**: 1건당 약 8-12원, 사용자당 월 30원 미만
- **장점**:
  - 법적 신뢰도 최고 (표준약관 인정 "문자")
  - 도달률 압도적 (이메일 20-30% vs 알림톡 90%+)
  - 한국 사용자 멘탈 모델 일치 (병원/택배/결제 알림 = 카톡)
  - PWA 푸시 제약 우회 (iOS Safari 16.4+ 필요, 권한 거부 시 무력화)
- **검수**: 카카오 비즈니스 채널 + 알림톡 템플릿 사전 승인 (1-3영업일)
- **연동**: NHN Toast / Aligo / 솔루션박스 또는 포트원 통합

### 알림 인박스 — 신규 컴포넌트
본인 코드에 `myFeedbackInbox`, `unreadBadge` 패턴 이미 있음 → **일반화**해서 알림 인박스 신설.

```
설정 → 알림함 (영구 기록)
  - 환영 선물 만료 7일 전 [읽음/안읽음]
  - 다음 결제 7일 전 [예정] (자동결제 도입 후)
  - 결제 영수증 [영구]
```

한 번 만들면 자동결제 도입 시 그대로 재활용.

## 7. 인증 — 카카오 OAuth Only

### 결정
- **신규 가입**: 카카오 OAuth만
- **기존 이메일 사용자**: 마이그레이션 (Supabase `linkIdentity`)
- **마이그레이션 완료 후**: 이메일 로그인 코드 제거
- **유예 기간**: 본인 직접 안내, 모두 완료 확인 후 제거 (10명 규모)

### 근거

**(a) UX 압승.** 한국 이메일+비밀번호 가입 마찰 큼. 카톡 OAuth는 2-tap. 가입 전환율 30-50% 상승 (한국 앱 일반).

**(b) 알림톡 채널 자연 통합.** 로그인 = 카톡이면 알림톡 발송 권한 자동 확보.

**(c) 어뷰저 차단.** 일회용 이메일 가입 불가. 카카오 계정 ≈ 1:1 with 실제 사람.

**(d) 미성년자 판별.** 카카오 OAuth로 출생연도 받아옴 → `requiresLegalGuardianForPayment` flag 자동 처리. 별도 본인인증 X.

**(e) 새 브라우저 진입 마법.** 카톡 데스크톱 로그인되어 있으면 새 PC에서 1-2탭 → 모든 데이터 동기화. 이메일/비밀번호보다 *더* 안정적.

### 카카오 OAuth scope (확정)
- **이메일** (선택적 — 카톡 가입 방식에 따라 없을 수 있음)
- **출생연도** (미성년자 판별용)
- 닉네임 / 프로필 사진 X

### 새 브라우저 진입 흐름

```
[새 브라우저에서 soragodong.com 열림]
  ↓
1. 첫 화면: [카카오로 시작하기] 한 버튼
  ↓
2. 카카오 OAuth 완료 → Supabase가 카카오 ID 검색:
   - 이미 link된 user 있음 → 자동 로그인 ✅ (기존 데이터 즉시 동기화)
   - 처음 보는 카카오 ID → 분기 모달 (다음 단계)
  ↓
3. 분기 모달:
   ┌───────────────────────────────────┐
   │  소라고동에 처음이야?               │
   │                                    │
   │  [응, 새로 시작할래]                 │
   │   → 신규 가입 + 튜토리얼              │
   │                                    │
   │  [아니, 예전 계정이 있어]             │
   │   → "카톡 오픈채팅으로 김나은에게 문의" │
   │     본인이 admin 도구로 수동 link     │
   └───────────────────────────────────┘
```

10명 규모는 자동화의 가치 < 단순함의 가치. 본인이 직접 처리 가능.

### E2EE 사용자 처리
- 새 브라우저 로그인 → auth 성공 → 데이터 fetch (암호화 상태) → E2EE 비밀번호 입력 → 복호화
- *의도된 설계*. 카톡 OAuth 마이그레이션도 E2EE 키 안 건드림 (인증 레이어와 데이터 레이어 분리).

### 마이그레이션 흐름 (기존 10명)

**Phase 1: 카톡 OAuth 추가 (1-2주)**
- 코드에 카카오 로그인 버튼 추가
- 신규 사용자는 카톡 only로 가입
- 기존 이메일 사용자는 *기존 방식 + 카톡 연결 옵션* 둘 다 가능

**Phase 2: 기존 10명 마이그레이션 (1-2주)**
- 본인이 카톡으로 한 명 한 명 안내
- 사용자 앱 진입 시 모달:
  > "카카오 계정 연결할까? 다음 업데이트부터 카톡 로그인만 가능해져."
- Supabase `linkIdentity({ provider: 'kakao' })` 호출
- **이메일 일치**: 자동 link → 알림 "카카오와 연결됐어"
- **이메일 불일치 / 카톡에 이메일 없음**: 본인 확인 모달 (옵션 C)

**Phase 3: 이메일 로그인 제거 (Phase 2 완료 후)**
- 모두 카톡 연결 완료 확인
- 이메일/비밀번호 가입·로그인 UI 제거
- Supabase의 이메일 provider 비활성화

### 본인 확인 모달 (옵션 C — 단순 체크박스)

```
┌───────────────────────────────────┐
│  잠깐, 본인 확인이 필요해           │
│                                    │
│  현재 앱 계정:                      │
│  📧 n***@gmail.com                  │
│                                    │
│  연결할 카카오 계정:                 │
│  📧 카카오 이메일 없음               │
│  (또는: k***@naver.com)             │
│                                    │
│  본인이 맞고, 두 계정 모두 본인 거    │
│  맞아? 잘못 연결되면 데이터 접근      │
│  못 하게 될 수 있어.                 │
│                                    │
│  [ ] 응, 본인 맞아                  │
│  [연결하기 (체크 시 활성)]  [취소]    │
└───────────────────────────────────┘
```

**선택 근거 (옵션 C)**:
- 10명 규모 + 본인이 직접 카톡으로 안내 → 본인이 fail-safe 역할
- 이메일 인증 코드 (옵션 A) 같은 강한 보호는 100명+ 단계에서 도입
- PWA + 카톡 minimum scope (이메일/출생연도만) → 닉네임 표시 불가, 단순 모달이 자연스러움

### 코드 변경 범위

| 영역 | 변경 정도 |
|---|---|
| 가입 화면 | 재설계 — 카카오 OAuth one-button only |
| 로그인 화면 | 카카오 버튼, 기존 이메일 로그인은 유예 동안만 별도 메뉴 |
| 마이그레이션 모달 | 신규 컴포넌트, 기존 사용자 첫 로그인 시 1회 노출 |
| `signupSession`, `authUserId` | Supabase `linkIdentity` 호출 추가 |
| 백엔드 API (`/api/billing/*`) | 변경 X (Supabase 토큰 그대로) |
| `soragodong_data` 테이블 | 변경 X (auth_user_id 그대로) |
| 분기 모달 (신규/기존) | 신규 컴포넌트 |
| Admin 도구 | 수동 link 도구 추가 (이메일 불일치 케이스 처리용) |

**핵심**: Supabase 인증 레이어만 재설계, 데이터 레이어는 안 건드림.
